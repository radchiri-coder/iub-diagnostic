# ТЗ v2: «Индекс Управляемости Бизнеса»

## 1. Цель документа

Этот документ описывает доработку локальной MVP-версии продукта «Индекс Управляемости Бизнеса» так, чтобы:

- проверить механику продукта локально до запуска на реальных пользователях;
- получить качественную, модульную и расширяемую кодовую базу;
- убрать основные логические дефекты первой версии;
- подготовить архитектуру для последующего переноса на сервер без переписывания ядра;
- заложить основу для персонализации по контексту компании, включая отрасль.

Важно: на этом этапе локальная версия нужна **не для реального сбора лидов**, а для тестирования сценария, UX, логики вопросов, скоринга, интерпретации и устойчивости архитектуры. Поэтому отсутствие серверной отправки данных на этой фазе допустимо и является частью требований.

---

## 2. Продуктовая цель

Нужно создать одностраничное веб-приложение на React + TypeScript + Tailwind CSS, которое проводит экспресс-диагностику управляемости компании малого и среднего бизнеса.

Пользователь проходит:

1. экран входа;
2. экран контекста компании;
3. блок из 21 вопроса;
4. локальный контактный гейт;
5. экран результатов.

На выходе пользователь получает:

- общий индекс управляемости;
- разбивку по 6 управленческим категориям;
- интерпретации по категориям;
- сравнительный ориентир по похожим компаниям;
- блок «Главный рычаг»;
- предупреждения по расхождениям прямых и косвенных индикаторов.

---

## 3. Главные принципы реализации

### 3.1. Принципы продукта

- Это не психометрический тест и не «оценка личности».
- Это экспертный диагностический инструмент для управленческой рефлексии.
- Результат должен выглядеть как профессиональный управленческий разбор, а не как развлекательный квиз.
- Пользователь не должен чувствовать, что его «манипулятивно ведут» к продаже.
- Даже в локальной версии продукт должен создавать ощущение экспертности, ясности и доверия.

### 3.2. Принципы архитектуры

- Контент и логика должны быть максимально отделены.
- Скоринг и анализ должны быть реализованы как чистые функции.
- UI не должен содержать бизнес-логику.
- Хранение данных должно быть вынесено в адаптеры.
- Все сущности должны быть типизированы.
- Перенос на сервер должен требовать замены адаптеров, а не переписывания ядра.

### 3.3. Принципы MVP

- Локальная версия тестирует механику.
- Никаких реальных внешних запросов в MVP.
- Контактные данные не отправляются никуда.
- Все данные живут только в браузере пользователя.
- При этом структура данных и сервисный слой должны быть готовы к последующему переходу на API.

## 4. Объем работ

Нужно не «подлатать» существующий `Index.tsx`, а собрать полноценную структурированную версию приложения с нормальным разделением ответственности.

Текущее состояние:

- есть сырой каркас на `useReducer`;
- есть базовый flow экранов;
- есть `localStorage`;
- отсутствует модульность;
- отсутствует качественная модель данных;
- нет зрелой логики скоринга и интерпретации;
- не решены риски социальной желательности;
- отсутствует нормальная аналитическая модель;
- отсутствует слой подготовки к серверной версии.

### 5.5. Шаг 5. Results

Пользователь получает:

- общий балл;
- радар или mobile-fallback;
- сравнительный ориентир;
- интерпретации по 6 категориям;
- главный рычаг;
- предупреждения по расхождениям;
- CTA-блоки как UX-заглушки для будущей боевой версии.

## 11. Контекст компании

### 11.1. Размер компании

Минимум 4 варианта:

- `up_to_15`
- `15_to_50`
- `50_to_150`
- `150_plus`

### 11.2. Стадия компании

Минимум 4 варианта:

- `survival`
- `stabilization`
- `growth`
- `rebuild`

### 11.3. Отрасль

Поле отрасли оставляем в MVP как обязательную часть архитектуры.

Причина:

- это важно для будущей сегментации;
- это важно для накопления собственной базы;
- это важно для сравнения с похожими компаниями;
- это потенциально важно для вариативности вопросов и интерпретаций;
- если убрать отрасль сейчас, потом будет больно перестраивать структуру данных и аналитику.

### 11.4. Но как именно использовать отрасль в MVP

На локальном MVP этапе отрасль **не должна менять базовый скоринг**.

Она должна использоваться для:

- сохранения в сессии;
- аналитики;
- подготовки будущих отраслевых бенчмарков;
- выбора отраслевой версии интерпретации, если она есть;
- выбора отраслевого текста сравнения, если он доступен;
- поддержки механизма `industryOverrides` в вопросах.

### 11.5. Важное ограничение по отрасли

Не нужно сейчас делать отдельные полные банки вопросов для каждой индустрии, если для них нет методологически выверенного контента.

Нужно сделать архитектуру, которая поддерживает:

- общий базовый банк вопросов;
- точечные отраслевые оверрайды для отдельных вопросов;
- отраслевые интерпретации;
- отраслевые бенчмарки;
- масштабирование в будущем.

### 11.6. Структура отраслевых оверрайдов

В `Question` предусмотреть:

- `industryOverrides?: Partial<Record<IndustryId, IndustryQuestionOverride>>`

Где `IndustryQuestionOverride` может менять:

- формулировку вопроса;
- текст вариантов;
- примеры;
- пояснение;
- но не ломать базовую шкалу.

## 12. Вопросы: методологические требования

Нужно 21 вопрос:

- 18 основных;
- 3 контрольных.

### 12.1. Структура категорий

6 категорий:

- Ясность курса
- Принятие решений
- Доведение до результата
- Структура и роли
- Командное взаимодействие
- Работа с людьми

На каждую категорию:

- по 3 основных вопроса.

Контрольные вопросы:

- 3 штуки;
- каждый проверяет пару категорий или одну широкую связку индикаторов.

### 12.2. Главная проблема первой версии, которую надо исправить

Вопросы не должны позволять пользователю слишком легко угадать “лучший” ответ.

### 12.3. Как именно снижать социальную желательность

Нужно использовать все механики вместе:

1. Ситуационные формулировки вместо абстрактных оценочных вопросов.
2. Варианты ответа, каждый из которых звучит жизненно и правдоподобно.
3. У части вопросов “сильный” ответ не должен звучать самым «красивым».
4. Должны быть 3-4 “ловушки”, где ответ звучит зрело, но на деле отражает ручное управление или незрелый процесс.
5. Порядок ответов должен рандомизироваться при первом показе вопроса.
6. Этот порядок должен фиксироваться в сессии и не меняться при возврате назад.
7. Нельзя строить весь банк вопросов по одному и тому же паттерну.

### 12.4. Требования к формулировкам

Запрещено:

- «Есть ли у вас стратегия?»
- «Насколько у вас сильная команда?»
- «Хорошо ли у вас выстроены процессы?»

Нужно:

- конкретная рабочая сцена;
- наблюдаемое поведение;
- узнаваемая ситуация;
- практический контекст;
- минимум оценочности.

### 12.5. Требования к вариантам ответов

Каждый вопрос должен иметь 4 варианта.

Для каждого варианта:

- однозначный балл;
- правдоподобный текст;
- естественный тон;
- отсутствие слишком явного “идеального ответа”.

### 12.6. Рандомизация ответов

Для каждого вопроса:

- сохранять `displayOrder`;
- при первом рендере перемешать, если `shuffleOptions = true`;
- при повторном показе брать сохраненный порядок;
- аналитика должна знать исходный порядок и порядок показа.

### 12.7. Контрольные вопросы

Контрольные вопросы нужны не для “наказания”, а для проверки расхождения между:

- прямыми самооценками;
- косвенными индикаторами зрелости.

Они не должны доминировать в общем балле.

Они должны использоваться для:

- предупреждений;
- меток уверенности в результате;
- внутренней аналитики.

### 12.8. Обязательные поля в `questions.ts`

У каждого вопроса должны быть:

- `id`
- `kind`
- `categoryId`
- `controlForCategoryIds?`
- `prompt`
- `options`
- `shuffleOptions`
- `tags`
- `industryOverrides?`

У каждого ответа:

- `id`
- `label`
- `score`
- `isTrap?`

## 14. Скоринг: базовые принципы

### 14.1. Ключевое правило

Контекст компании **не должен менять сырой балл ответа**.

То есть:

- пользователь выбрал ответ;
- у ответа есть базовый вес;
- этот вес участвует в расчете категории;
- категория участвует в общем индексе.

Размер, стадия и отрасль влияют не на сам факт “насколько зрелым был ответ”, а на:

- интерпретацию;
- силу риска;
- ожидаемую норму;
- сравнительный ориентир;
- приоритетность рекомендации;
- выбор “главного рычага”.

### 14.2. Почему это важно

Если контекст меняет сами баллы, результат становится:

- менее прозрачным;
- менее объяснимым;
- более похожим на “черный ящик”;
- менее доверительным для экспертного продукта.

## 15. Логика расчета результата

Реализовать в `engine/scoring.ts` и `engine/analysis.ts`.

### 15.1. Расчет категории

`calculateCategoryScore(answers, categoryId)`

Должно:

- брать только основные вопросы категории;
- агрегировать их баллы;
- нормализовать на шкалу 0–10;
- возвращать предсказуемый результат.

### 15.2. Расчет общего балла

`calculateOverallScore(categoryScores)`

Должно:

- агрегировать 6 категорий;
- возвращать число на шкале 0–10;
- быть устойчивым к отсутствию части данных только при незавершенной сессии;
- в финальном результате работать только для завершенного прохождения.

### 15.3. Контрольные расхождения

`detectControlDiscrepancies(answers, categoryResults)`

Должно:

- сравнивать контрольные сигналы с релевантными категориями;
- считать расхождение;
- формировать предупреждения при дельте выше порога.

Порог на MVP:

- 3 балла по шкале 0–10.

### 15.4. Уровни результата

Для общего балла и категорий ввести уровни:

- `high`
- `medium`
- `low`

Пример стартовой градации:

- `7.5–10` → high
- `5.0–7.4` → medium
- `<5.0` → low

Сделать это конфигурируемым, а не захардкоженным в UI.

### 15.5. Главный рычаг

`findTopLeverage(categoryResults, context)`

Должен находить не просто самую слабую категорию, а категорию с наилучшим сочетанием:

- текущего дефицита;
- влияния на систему;
- релевантности стадии компании;
- ожидаемого практического ROI.

Для MVP ввести прозрачную формулу:

`leverageScore = gapWeight * gapToStrongLevel + stagePriorityWeight * stagePriority + dependencyWeight * systemicDependency`

Где:

- `gapToStrongLevel` показывает, насколько категория недотягивает до сильного уровня;
- `stagePriority` задается по стадии компании;
- `systemicDependency` задается экспертно в конфиге.

Нужно обязательно сделать эту формулу явной и комментированной в коде.

## 16. Интерпретации и персонализация

Реализовать в `config/interpretations.ts` и `engine/personalization.ts`.

### 16.1. Базовая матрица

Нужна матрица:

- 6 категорий
- 3 уровня
- 4 стадии компании

Минимум:

- 72 базовых интерпретационных блока.

### 16.2. Что должно быть в каждом блоке

Каждый блок интерпретации должен содержать:

- краткий диагноз;
- что это означает для компании на этой стадии;
- главный риск;
- первый практический шаг.

### 16.3. Отраслевые расширения

В архитектуре предусмотреть возможность отраслевого уточнения интерпретации.

Не обязательно заполнить отраслевые версии для всего массива сейчас, но механизм должен быть:

- если есть отраслевой override, использовать его;
- если нет, использовать базовую интерпретацию.

### 16.4. Блок «Главный рычаг»

Нужен отдельный генератор:

`getLeverageRecommendation(categoryResults, context)`

Он должен возвращать:

- категорию;
- короткое объяснение, почему именно она сейчас ключевая;
- 1 первый шаг;
- 1 ожидаемый эффект.

### 16.5. Сравнение с похожими компаниями

`compareToBenchmark(results, context)`

Должно:

- сравнивать результат с ориентиром по размеру, стадии и, если доступно, отрасли;
- возвращать понятную дельту;
- честно маркировать источник.

Важно:

пока нет настоящей статистической базы, **нельзя писать “по реальным данным рынка”**.

Формулировка для MVP:

- `Экспертный ориентир для компаний вашего профиля`
- `Предварительный ориентир, основанный на методологии и накопленных кейсах`

Не использовать формулировки, создающие ложное впечатление строгой репрезентативной выборки.

## 18. Контактный гейт

### 18.1. Цель на MVP

Проверить механику:

- в какой момент просить контакт;
- какой текст лучше воспринимается;
- как соотносится ценность до и после гейта;
- не раздражает ли этот шаг пользователя.

### 18.2. Поведение

Пользователь видит:

- заголовок;
- пояснение;
- поля для `email` и/или `telegram`;
- кнопку продолжить;
- явную возможность пропустить.

### 18.3. Важно для доверия

Нельзя строить UX как шантаж ценностью.

Нужно:

- при пропуске показывать полезный, но более базовый результат;
- при вводе контакта показывать расширенные текстовые блоки.

### 18.4. Разделение уровней результата

Если пользователь пропустил гейт:

- показать общий балл;
- показать визуализацию;
- показать 6 категорий с краткими статусами;
- скрыть расширенные интерпретации и блок детального следующего шага.

Если пользователь заполнил гейт:

- показать полный текст;
- показать главный рычаг;
- показать расширенные рекомендации.

### 18.5. Локальная логика

Ввод:

- валидируется;
- хранится только локально;
- не отправляется наружу.

Нужен явный текст или микрокопи, который не вводит в заблуждение на локальной демо-версии.

## 19. Хранение данных

Реализовать через сервисный слой и `usePersistence`.

### 19.1. Требования

- вся сессия хранится как версия схемы;
- данные восстанавливаются после перезагрузки;
- при изменении структуры должна работать миграция;
- порядок вопросов и порядок ответов сохраняются;
- хранение результата должно быть отделено от хранения контакта.

### 19.2. Формат

```ts
{
  version: 2,
  data: {
    session: ...,
    analytics: ...,
    ui: ...
  }
}
```

### 19.3. TTL

Добавить срок жизни локальной сессии:

- по умолчанию 30 дней.

Если срок истек:

- старая сессия мягко сбрасывается;
- пользователь начинает заново.

### 19.4. Миграции

Нужно реализовать:

- `migratePersistedState(fromVersion, toVersion)`

Минимум:

- защита от падения на старой схеме;
- graceful fallback на новый `initialState`.

## 21. Безопасность и работа с данными

### 21.1. На MVP обязательно

- никаких данных в URL;
- никакой отправки вовне;
- валидация email;
- очистка/trim пользовательского ввода;
- не хранить больше данных, чем нужно;
- возможность сбросить локальный прогресс.

### 21.2. Контактные данные

Контакт хранить отдельно от результата.

Нельзя:

- смешивать `lead` и `answers` в одну неструктурированную кучу;
- использовать контакт как часть идентификатора сессии.

### 21.3. CSP

Добавить разумную CSP для статической локальной версии, не ломая загрузку шрифтов и локального бандла.

### 21.4. Псевдонаучность

Обязательно добавить дисклеймер:

- это не психометрический тест;
- это экспертный инструмент интерпретации управленческих паттернов;
- результаты полезны как карта для разговора и приоритизации, а не как абсолютный диагноз.

## 34. Что особенно важно не потерять

Это не просто техническая задача. Нужно сохранить смысл продукта:

- экспертность;
- доверие;
- интеллектуальную глубину;
- ощущение точного управленческого инструмента;
- возможность дальнейшего роста в полноценный цифровой диагностический продукт.

Главная цель этой итерации: не “сделать еще один квиз”, а собрать устойчивый каркас действительно качественного экспертного продукта, который можно безопасно тестировать локально, а потом переносить в боевую среду почти без смены ядра.
