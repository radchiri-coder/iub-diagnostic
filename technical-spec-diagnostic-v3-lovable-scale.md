# ТЗ v3: «Индекс Управляемости Бизнеса»
## Локальный MVP в Lovable + архитектура роста до Gallup-подобной платформы

## 1. Зачем этот документ

Этот документ объединяет:

- финальные требования к локальному MVP;
- требования к сборке проекта так, чтобы его было удобно запускать через Lovable;
- архитектурные требования для последующего переноса на боевую серверную версию;
- целевое направление развития в сторону платформы уровня организационной диагностики:
  - результаты компании целиком;
  - результаты по отделам;
  - сравнение с предыдущими периодами;
  - сравнение разных уровней управления;
  - разная глубина доступа для собственника, руководителя и сотрудника.

Главная идея:

- **MVP делаем простым для запуска и тестирования**;
- **ядро делаем взрослым и переносимым**;
- **данные и модель сразу проектируем так, чтобы потом не переписывать продукт с нуля**.

---

## 2. Короткий вывод по стеку

### 2.1. Рекомендуемое решение

Да, **Lovable как среда для первой рабочей версии — хорошее решение**, если соблюдены 4 правила:

1. Код с первого дня подключен к GitHub.
2. Бизнес-логика не зашита в визуальные компоненты.
3. Данные и авторизация в боевой версии живут не “внутри Lovable”, а в отдельном бекенде, оптимально в Supabase/Postgres.
4. Продукт проектируется не как одноразовый квиз, а как система с доменной моделью.

### 2.2. Когда это хорошее решение

Lovable хорош для:

- быстрого вывода первой usable-версии;
- визуальной сборки UI;
- быстрых итераций по текстам, flow и интерфейсу;
- совместной работы с неразработческим способом мышления;
- быстрого опубликования демо без долгого локального разворачивания.

### 2.3. Когда это плохое решение

Lovable будет плохим решением, если:

- проект не синхронизируется с GitHub;
- вся логика останется на уровне промптов и generated-компонентов;
- структура данных не будет спроектирована заранее;
- доступы, роли, агрегаты и организационная аналитика будут “додумываться потом”.

### 2.4. Итоговая рекомендация

Для этого продукта оптимален путь:

1. Собрать архитектурно правильный фронтенд и доменную модель.
2. Открыть проект в Lovable для удобной сборки, публикации и визуальных итераций.
3. Держать код в GitHub как источник правды.
4. Когда MVP подтвердит механику, подключить Supabase и перенести хранилище, пользователей, роли и организационные данные на Postgres.

---

## 3. На что опирается рекомендация

По актуальной документации Lovable:

- Lovable поддерживает просмотр и ручное редактирование кода в editor/code mode.
- Lovable синхронизируется с GitHub, и код можно использовать локально и вне платформы.
- Lovable прямо заявляет полное владение кодом и возможность экспорта.
- При подключении Supabase данные остаются в portable PostgreSQL-инфраструктуре и не запираются внутри Lovable.

Поэтому риск vendor lock-in здесь **умеренный, а не критичный**, если:

- проект подключен к GitHub;
- база отдельно;
- схемы и RLS-контуры находятся на стороне Supabase/Postgres.

## 5. Продуктовая рамка

## 5.1. Что это за продукт сейчас

Сейчас это:

- экспертный диагностический инструмент;
- B2B/B2B2C продукт для собственников и управленцев;
- сначала личная диагностика;
- затем организационная диагностика;
- затем аналитическая платформа управляемости компании.

## 5.2. Целевая эволюция продукта

### Этап A. Локальный MVP

- один пользователь;
- 21 вопрос;
- локальная сессия;
- локальный контактный гейт;
- результаты по 6 категориям;
- без реальной базы и без реальных пользователей.

### Этап B. Боевая индивидуальная версия

- реальный запуск;
- база ответов;
- регистрация или magic link;
- сохранение сессий;
- история прохождений;
- бенчмарки на реальных данных;
- лиды и заявки.

### Этап C. Командная версия

- сотрудники компании тоже проходят диагностику;
- есть привязка к компании и отделу;
- есть уровни доступа;
- можно видеть компанию целиком и по подразделениям;
- можно сравнивать собственника и менеджмент;
- можно сравнивать волну 1 и волну 2.

### Этап D. Платформа уровня Gallup-like

- организационный дашборд;
- агрегаты по отделам, ролям, уровням;
- сравнение во времени;
- сегментация по функциям;
- разные уровни глубины для разных ролей;
- организационная карта управляемости;
- безопасный ролевой доступ к данным.

## 6. Главный принцип проектирования

Нужно делать **не квиз**, а **ядро диагностической платформы**, у которой первая версия просто визуально выглядит как опросник.

Отсюда следуют 3 правила:

1. Вопросы, ответы, категории и интерпретации должны быть отдельной моделью данных.
2. Индивидуальный результат должен быть построен так, чтобы потом его можно было агрегировать в командный и организационный уровень.
3. Данные должны храниться так, чтобы потом можно было считать:
   - средние по компании;
   - средние по отделу;
   - различия по ролям;
   - тренды во времени;
   - расхождения между самооценкой руководителя и оценкой сотрудников.

## 7. Что строим прямо сейчас

Нужно создать **одностраничное React/TypeScript/Tailwind приложение**, которое:

- запускается как обычный фронтенд-проект;
- может быть загружено и дорабатываться в Lovable;
- имеет чистую архитектуру;
- использует локальное хранилище на MVP;
- не зависит от установленного локально Node.js на этапе передачи/согласования;
- может быть потом развернуто через Lovable publish или локально после установки Node.js.

## 8. Вывод по Lovable как способу запуска

### 8.1. Для MVP

Да, запускать первую рабочую версию через Lovable — **разумно**.

Плюсы:

- меньше технического трения;
- быстрее показать живой продукт;
- проще править внешний вид;
- проще публиковать демо;
- проще править тексты и flow;
- удобно, если не хочется самой долго разворачивать среду.

### 8.2. Для долгосрочной платформы

Lovable не должен быть единственным “местом жизни продукта”.

Правильный контур:

- **Lovable = среда сборки и публикации интерфейса**
- **GitHub = источник правды по коду**
- **Supabase/Postgres = источник правды по данным**

Тогда переносимость будет хорошей.

### 8.3. Итог

Да, **это хорошее решение**, если сразу потребовать:

- GitHub sync;
- нормальную файловую структуру;
- отдельный data-model слой;
- отдельный storage/service слой;
- отсутствие “магии только в Lovable”.

## 10. Уровни доступа будущего продукта

Нужно проектировать продукт с будущей ролевой моделью.

### 10.1. Базовые роли будущей серверной версии

- `owner`
- `executive`
- `manager`
- `employee`
- `consultant_admin`
- `consultant_analyst`

### 10.2. Что будет видеть каждая роль

#### Owner

- общий результат компании;
- результаты по отделам;
- динамику по периодам;
- карту рисков;
- сравнительный контур по управленческим уровням;
- возможно, агрегированный снимок по людям без излишней персональной детализации.

#### Executive / Top manager

- результаты своих функций/блоков;
- результаты смежных команд в ограниченном виде;
- тренды;
- рекомендации.

#### Manager

- данные своего подразделения;
- динамика по команде;
- сравнение с общей компанией в ограниченном виде;
- свои собственные персональные результаты.

#### Employee

- только собственный результат;
- возможно, обезличенный контекст команды;
- без доступа к персональным данным коллег.

#### Consultant roles

- доступ по назначенным организациям;
- возможность видеть агрегаты;
- возможность вести историю волн;
- экспорт аналитики и отчеты.

## 11. Принцип глубины результатов

Да, идея “как у Gallup” здесь очень уместна.

Нужно поддержать принцип:

- **одни и те же данные -> разные уровни представления в зависимости от роли**

Это значит:

- у сотрудника — личный результат;
- у руководителя — личный + агрегат команды;
- у собственника — личный + компания + отделы + динамика;
- у консультанта — диагностическая аналитика верхнего уровня.

Это не нужно полноценно реализовывать в локальном MVP, но **обязательно** нужно учесть в data model и серверной дорожной карте.

## 5.1. Главное правило скоринга

**Контекст не меняет сырой балл ответа.**

Это значит:

- score ответа фиксирован;
- score категории считается из ответов;
- общий score считается из категорий;
- контекст влияет только на:
  - интерпретацию;
  - ориентир;
  - приоритет;
  - главный рычаг;

Это обязательное правило.

- flow экранов;
- расчет derived state;
- запуск аналитики;
- переходы между шагами;
- итоговый state machine.

### 7.2. `usePersistence`

Отвечает за:

- сохранение;
- восстановление;
- версию схемы;
- миграции;
- TTL;
- reset.

### 7.3. `useAnalytics`

Отвечает за:

- типизацию событий;
- отправку в адаптер;
- временные метки;
- сессии.

### 7.4. `useQuestionFlow`

Отвечает за:

- порядок вопросов;
- стабильный `displayOrder` ответов;
- возврат назад;
- время на вопрос;
- определение текущего шага.

### 7.5. `useViewport`

Для адаптивного переключения:

- радар;
- category bars.

## 17. Persistence и версия схемы

Формат:

```ts
{
  version: 3,
  data: {
    session: ...,
    ui: ...,
    analytics: ...,
    lead: ...
  }

```

- TTL 30 дней;
- graceful reset;
- защита от битого состояния;

## 20. Требования к Lovable-совместимости

Проект должен быть удобен для дальнейшей загрузки/работы в Lovable.

Поэтому:

- структура должна быть обычной для React/Vite;
- Tailwind-конфиг должен быть прозрачным;
- не использовать слишком экзотические build-хаки;
- не строить архитектуру вокруг локальных костылей;
- все должно нормально жить в GitHub;
- не делать монолит в одном файле.

Нужно считать, что проект будет:

1. загружен/синхронизирован через GitHub;
2. дорабатываться визуально и текстово в Lovable;
3. публиковаться как demo;
4. потом переноситься на прод-контур.

---

## 21. Что важно для легкого переноса дальше

Перенос будет **относительно легким**, если с самого начала выполнить эти условия:

1. Бизнес-логика остается в `engine`.
2. Данные и тексты остаются в `config`.
3. Хранение идет через `services`.
4. UI ничего не знает о том, откуда пришли данные.
5. Индивидуальный результат строится в форме, пригодной для будущего агрегирования.

Если это соблюдено, переход к серверной версии будет не “переписыванием фронта”, а:

- заменой local adapters на API adapters;
- добавлением авторизации;
- добавлением сущностей организации, команды и участников;
- добавлением серверных запросов и дашбордов.

## 26. Acceptance Criteria для текущего этапа

Работа считается выполненной, если:

- создан новый проект с нуля;
- он совместим с React/Vite/Tailwind и годится для Lovable;

- есть 5 экранов;
- есть 21 вопрос;

- реализована стабильная рандомизация ответов;
- порядок ответов не меняется при возврате назад;
- есть 72+ интерпретации;
- есть экспертные ориентиры;
- есть локальный гейт;
- есть базовый и полный режим результата;

- есть контрольные расхождения при дельте >= 3;
- есть версия схемы;
- есть миграции;
- есть TTL;

## 27. Обязательная верификация

Проверить:

1. Полное прохождение всех 21 вопроса.
2. Возврат назад не меняет порядок ответов.
3. Перезагрузка страницы восстанавливает прогресс.
4. Битый `localStorage` не ломает приложение.
5. Истечение TTL мягко сбрасывает старую сессию.
6. Пропуск гейта показывает только базовый результат.
7. Заполнение гейта открывает полный результат.
8. На мобильном используются бары вместо основного радара.
9. Контрольные предупреждения показываются при дельте >= 3.
10. Аналитические события пишутся локально.
11. Нет сетевых запросов.
12. Длинные тексты и длинные названия отраслей не ломают layout.
